
# 🕹️ SKILL — 游戏界面易用性分析 (UI Usability Analyzer)

> 对游戏截图进行系统化易用性/可用性诊断，从热区尺寸、费茨定律、误触风险、认知负荷、视野遮挡等7个维度量化输出问题列表与优化建议，驱动有数据支撑的 UI 改进决策。

**模块归属**：`01_game_info / collectors`
**当前状态**：✅ Phase 1 可用（截图输入 + GPT-4o Vision 分析）
**脚本文件**：`project/ui_usability_analyzer/analyzer.py`

---

## 🎯 功能定位

输入一张游戏界面截图（可附加按键标注 JSON），自动完成：

1. **热区尺寸核查** — 检测各按键 px 尺寸，换算为 dp，对照平台触控标准（移动端 ≥44dp，PC/平板 ≥32dp）
2. **费茨定律评估** — 量化按键大小与操作距离，判断高频操作是否落在拇指热区
3. **热区间距风险** — 检测相邻按键间距是否 ≥20px，预防误触
4. **热区重叠检测** — 识别负间距/重叠区域，标记功能性可用性问题
5. **认知负荷评估** — 统计同屏元素数量，评估图标辨识度与视觉层级合理性
6. **视野遮挡分析** — 计算按键组占屏比，评估透明度策略与游戏内容遮挡情况
7. **拇指热区分布** — 判断各按键落在舒适区 / 可达区 / 困难区哪个操作层

输出 **HTML 可视化报告** + **JSON 结构化数据**，每条建议附带开发成本标注（L/M/H）。

## 📐 分析维度

| 维度 | 理论依据 | 具体检测内容 | 判定标准 |
|---|---|---|---|
| **热区尺寸** | ISO 9241-9 触控规范 / Apple HIG / Google Material | 各按键 px → dp 换算，与平台规范对照 | 移动端 ≥44dp；PC/平板 ≥32dp |
| **费茨定律** | Fitts's Law（1954）— 操作时间 = a + b·log₂(2D/W) | 按键大小与到达距离，高频操作在拇指热区内 | 高频按键位于屏幕下半 1/3 热区 |
| **热区间距** | Nielsen 误触风险原则 + 触控研究（Parhi et al.）| 相邻按键边缘间距（px）| 建议 ≥20px；警告 10–19px；危险 <10px |
| **热区重叠** | 可用性工程（Usability Engineering）| 负间距/按键边界相交检测 | 负间距为功能性 Bug 级别问题 |
| **认知负荷** | Cognitive Load Theory（Sweller 1988）+ 米勒定律（7±2）| 同屏元素数量、图标辨识度、视觉层级 | 元素数 ≤9 合理；>12 存在过载风险 |
| **视野遮挡** | 格式塔原则（连续性/图地关系）| 按键组占屏比（%），透明度策略合理性 | 单侧 UI 占屏比 ≤25% 为佳 |
| **拇指热区分布** | Steven Hoober 单手操作研究（2013）| 各按键归属：舒适区 / 可达区 / 困难区 | 高频操作应 ≥80% 落在舒适区 |

## 🛠️ 调用接口

```python
# 方式一：作为模块调用
from project.ui_usability_analyzer.analyzer import UIUsabilityAnalyzer

analyzer = UIUsabilityAnalyzer(
    image_path   = "resources/screenshots/ui_battle_001.jpg",
    resolution   = (1920, 1080),       # 屏幕分辨率，默认 1920×1080
    platform     = "mobile",           # "mobile" | "tablet" | "pc"
    button_data  = [                   # 可选：按键标注数据（来自设计稿/标注工具）
        {"name": "技能1",  "x": 1720, "y": 900, "width": 88, "height": 88},
        {"name": "普通攻击","x": 1820, "y": 980, "width": 96, "height": 96},
    ]
)

result = analyzer.run()
# result.report_html → HTML 报告路径
# result.data_json   → 结构化 JSON 数据
```

```bash
# 方式二：命令行调用
python analyzer.py \
  --image    screenshot.png \
  --resolution 1920x1080 \
  --platform mobile \
  --buttons  buttons.json \
  --output   ./reports/

# 查看帮助
python analyzer.py --help
```

**参数说明：**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `image_path` | string | ✅ | 截图本地路径（jpg/png/webp）|
| `resolution` | tuple | ❌ | 屏幕分辨率，默认 `(1920, 1080)` |
| `platform` | string | ❌ | 平台类型：`mobile`/`tablet`/`pc`，影响 dp 标准 |
| `button_data` | list[dict] | ❌ | 按键标注数据 JSON，含 name/x/y/width/height |
| `output_dir` | string | ❌ | 报告输出目录，默认 `./reports/` |
| `api_key` | string | ❌ | OpenAI API Key（优先从环境变量 `OPENAI_API_KEY` 读取）|

## ⚠️ 执行红线

1. **证据挂载**：所有尺寸/间距结论必须附上 `[截图编号 + 像素坐标]` 作为数据来源，严禁估算代替测量
2. **平台区分**：dp 标准因平台不同而异（移动端 ≥44dp / PC ≥32dp），禁止混用标准
3. **开发成本标注**：每条优化建议必须标注 `L（低）/ M（中）/ H（高）` 开发成本，不得只提问题不给成本评估
4. **重叠问题优先级**：热区负重叠/重叠属于功能性 Bug，报告中必须标为 P0，优先于体验优化问题
5. **GPT 分析局限性**：GPT-4o Vision 在精确像素坐标测量上存在误差，当传入 `button_data` 标注数据时，尺寸/间距计算以标注数据为准，Vision 结论仅作为辅助参考
