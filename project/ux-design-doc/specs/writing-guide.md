# 交互说明文档撰写规范

> 版本：v1.6 | 维护：喵喵 | 更新：2026-04-13

---

## 一、核心原则

### 1.1 三段式结构（v1.5）

每份交互文档固定三段：**文档头 → A. 需求范围 → B. 方案说明**。

> 完整格式示例见 `field-reference.md` §一。

**核心要点**：
- ✅ **文档头** — `icon` / `version` / `title` / `planner` / `uxd` 五行，文件顶部
- ✅ **需求范围** — `## A.` 固定，列出所有涉及界面名称
- ✅ **方案说明** — `## B.` 固定，按 `### [N]` 界面分块
- ✅ **界面内模块** — `#### N.` 从 1 起编，界面内局部编号
- ✅ **不跨界面连续编号** — 每个界面重新从 1 开始

### 1.2 何时用子模块

#### 模块拆分判断树（必须遵守）

> ⚠️ **判断依据是截图中的物理空间，不是"能不能单独命名"**

```
Q1：这个控件是否有独立的物理边界（自己占据独立区域、独立行/块）？
  → 否（它是某个卡片/行/区域的组成部分）
       └→ 归入父容器，声明为 **子控件：** 字段，不拆独立 ##### 块
  → 是 → Q2：它的交互逻辑是否可以完全独立理解（不依赖父容器状态）？
            → 否 → 仍归入父容器（**子控件：** 字段）
            → 是 → 独立 ##### 子模块
```

**典型正误对比**：

| 场景 | 错误做法 ❌ | 正确做法 ✅ |
|------|-----------|-----------|
| 成员卡片行内有「更多」按钮和「权限」下拉框 | 拆为三个独立 ##### 块 | 合并为一个「成员卡片」块，内部声明 `**子控件：** 更多操作按钮 / 下拉框` |
| 弹窗内有确认/取消两个按钮 | 每个按钮拆一个 ##### | 合并为「底部操作区」块，子控件声明 |
| 界面内有独立的搜索栏（占一整行） | 挂在其他模块下 | 独立 #### 块 |
| 卡片内展开/收起的子内容区 | 独立 #### 块 | 归入父卡片，在交互说明中描述展开逻辑 |

当一个模块内有**多个独立子控件**需要分别说明时，用 `#####` 内嵌子模块（不再用独立卡片）：

```markdown
#### 2. 套餐价格区
**类型：** 复合控件

1. 显示：
   a. 月价 / 年价 / 买断价

##### 1 折扣标签              ← 纯数字，无点，内嵌在父模块卡片
1. 有折扣：显示「省 ¥XX」
**控件状态：** 有折扣 / 隐藏

##### 2 立即订阅按钮
1. 已登录 → 调起支付页
**控件状态：** 常态 / 处理中
```

## 二、格式规范

### 2.1 层级格式

| 层级 | 格式 | 说明 |
|------|------|------|
| 界面块 | `### [N] 界面名称` | 方括号整数，B 节下 |
| 截图 | `design: 路径` | 界面块标题下一行 |
| 模块 | `#### N. 名称` | 界面内从 1 起编 |
| 子模块 | `##### N 名称` | 纯数字无点，嵌在父模块下 |

> 正误示例及 HTML 样式见 `field-reference.md` §2.2

### 2.2 字段顺序

推荐顺序：
```markdown
#### N. 控件名称
**类型：** xxx

1. 显示：
   a. xxx
   b. xxx（点击 → yyy）      ← 简单交互可内联
   【需要图示】               ← 可选

2. 交互说明：
   a. 点击【xxx】→ yyy
   b. 触发条件：xxx时触发     ← 可选，被动出现的控件填写；位置不固定，排在相关交互行为之后即可
   c. 消失条件：xxx时消失     ← 可选；位置同上
   【需要交互图示，热区 ≥ 80px】   ← 可选，多热区时加

**控件状态：** 常态 / 按下态 / 禁用态
**滑动热区：** xxx（如涉及滑动）
**@策划** xxx问题（如需要确认）
```

> 💡 **简单模块合并写法**：子项 ≤ 4 个且每项交互简单时，直接在 `1. 显示：` 内联写交互，省略 `2. 交互说明：`。

### 2.3 渲染顺序原则

> **你在 MD 里写的顺序，就是 HTML 里显示的顺序。解析器不重排任何字段。**

这意味着：

- **`**@职能**` 放在哪，HTML 里就出现在哪**。如果放在「控件状态」前，渲染后也在控件状态前。
- **子模块（`#####`）统一渲染在父模块全部字段之后**，与 MD 书写位置无关（这是唯一的例外）。
- 字段推荐顺序（`field-reference.md` §2.3）只是规范建议，不强制，但乱序会影响可读性。

**实际推荐书写顺序**：
```
**类型：**
1. 显示：
2. 交互说明：
**控件状态：**
**滑动热区：**（如有）
**空态：**（如有）
**@策划** / **@GUI** / **@程序**（如有）
##### N 子模块（如有）
```

### 2.4 控件状态写法

**基础格式**：`/` 分隔
```markdown
**控件状态：** 常态 / 按下态 / 禁用态
```

**多个控件的写法（推荐）**：
当模块包含多个独立控件时，使用 `控件名-状态名` 格式：
```markdown
**控件状态：** 山头条目-未选中态 / 山头条目-选中态 / 创建按钮-常态 / 创建按钮-按下态
```

> 渲染效果（HTML 样式细节）见 `field-reference.md` 字段4。

**红点 / 角标 / 标签**：作为独立状态列出，不与其他状态组合：
- ✅ `**控件状态：** 常态 / 按下 / 选中 / 红点`
- ❌ `**控件状态：** 常态+红点 / 按下+红点 / 选中+红点`

**避免**：`- ` 列表（会导致渲染错误）
```markdown
❌ **控件状态：**
- 山头条目：未选中态 / 选中态
- 创建按钮：常态 / 按下态
```

### 2.4 箭头使用

**只用于「交互行为 → 结果反馈」**：
- ✅ `点击 → 打开弹窗`
- ✅ `点击 → 提示toast<保存成功>`
- ✅ `b. 问号按钮（点击 → 打开通用说明弹窗）`（合并写法内联也适用）

**不用于状态描述**：
- ❌ `已拥有时 → 显示已拥有`
- ✅ `已拥有态：显示「已拥有」`

**无交互时省略 `2. 交互说明：` 整段**，不要写 `交互说明：无`：
- ❌ `2. 交互说明：无`
- ✅ （直接不写交互说明段落）

### 2.5 Toast 格式

必须用尖括号：`<toast内容>`，写在交互行末尾：
```markdown
✅ 点击 → 提示toast<模组已装载>
❌ 点击 → toast提示「模组已装载」」
```

> 完整格式规范（多场景写法 + 完整示例）见 `field-reference.md` §五。

## 三、术语统一

| 统一叫法 | 不用的叫法 |
|---------|-----------|
| 气泡 | 浮动标签、角标气泡、tooltip |
| 奖励道具框 | 奖励组件、复用奖励框 |
| 弹窗 | 弹框、popup、dialog |
| 点击 | 按下、tap |
| 页签 | tab、标签页 |
| 打开模组详情页 | 跳转模组详情页 |

> **跨文档命名不一致时**：交互稿与 PRD 对同一元素使用不同名称时（如「神秘方块」vs「幸运方块」），以**交互稿命名为准**，无需标注差异，不纠结。

## 四、需求缺漏分级

| 级别 | 含义 | 判断口诀 | 典型示例 |
|------|------|---------|---------|
| P0 | 流程断裂，无法继续 | 不处理能不能上线？→ 不能 | 网络异常无提示；关键按钮跳转未定义 |
| P1 | 影响体验，需明确处理 | 不处理用户会不会迷路/报错？→ 会 | 空列表无空态；多弹窗优先级未定义 |
| P2 | 边界情况，建议补充 | 不处理也能上线，但有点糙？→ P2 | 超长文本截断未定义；旧数据兼容展示 |

**格式**：
```markdown
## 需求缺漏

### P1
- **问题标题**
  问题描述

### P2
- **问题标题**
  问题描述
```

## 五、完整示例

```markdown
icon: 🌿
version: #A20483
title: 模组列表
planner: 林晓东
uxd: 喵喵

## A. 需求范围
- 模组列表主界面
- 操作确认弹窗

## B. 方案说明

### [1] 模组列表主界面
design: design/iface-list.png

#### 1. 模组信息区

**类型：** 信息展示区域

1. 显示：
   a. 模组图标（左侧，固定尺寸）
   b. 模组名称（主标题，单行）
   c. 模组类型标签（副标题，如「玩法扩展」）
   d. 状态标识（右侧，根据模组状态动态显示）

**控件状态：** 正常态 / 已装载态 / 已停用态 / 可更新态

#### 2. 操作按钮区

**类型：** 功能按钮组

1. 显示：
   a. 装载按钮（绿色）
   b. 卸下按钮（白色/灰色）

##### 1 装载按钮（绿色）
1. 显示「装载」
2. 点击 → 装载该模组
**控件状态：** 常态 / 按下态

##### 2 卸下按钮（白色/灰色）
1. 显示「卸下」
2. 点击 → 卸下该模组
**控件状态：** 常态 / 按下态

**控件状态：** 未装载态 / 已装载态 / 已停用态

## 需求缺漏

### P1
- **「已停用」触发条件未明确**
  什么情况下模组会被标记为「已停用」？

### P2
- **长按菜单是否支持**
  模组条目是否支持长按呼出更多操作？
```

## 六、交稿前检查清单

- [ ] 文档头五字段齐全（icon / version / title / planner / uxd）
- [ ] 包含 `## A. 需求范围` 和 `## B. 方案说明`
- [ ] 界面块格式正确（`### [N] 界面名称`）
- [ ] 模块标题格式正确（`#### 1. 名称`），界面内从 1 起编
- [ ] 没有用旧式子序号（`### N.N`），子模块改用 `##### N 名称`
- [ ] 控件状态用 `/` 分隔，不用 `- ` 列表
- [ ] Toast 用尖括号 `<内容>`
- [ ] 箭头只用于交互行为，不用于状态描述
- [ ] 每个模块都有内容（类型/显示/交互至少一项）

## 七、全局检查项（每个界面必过）

> 撰写完成后，逐项对照检查，未覆盖的记入「需求缺漏」。

### 货币与付费
- [ ] 需要付费的界面是否显示货币栏
- [ ] 货币种类与界面使用货币一致
- [ ] 限购商品：未购买 / 已购买状态
- [ ] 限时商品：未购买 / 已购买 / 过期状态
- [ ] 免费商品是否显示红点

### 领取相关
- [ ] 不可领取 / 可领取 / 已领取状态区分
- [ ] 点击不可领取&已领取 → Tips；可领取 → 获得弹窗
- [ ] 可领取状态是否显示红点
- [ ] 多个可领取项时是否有一键领取

### 空状态
- [ ] 所有列表是否有空态设计
- [ ] 空态是否引导跳转

### 加载状态
- [ ] 需要加载的内页是否有加载态

### 红点规范
- [ ] 红点链路完整（页签 → 子界面）
- [ ] 红点类型正确（常规 / 数字 / "新"标签）

### 动效
- [ ] 多段动效是否标注流程

### 奖励道具框识别
- [ ] 文案中凡涉及「道具」「奖励」等描述时，须对照截图判断是否为奖励道具框控件
- [ ] 如确认是奖励道具框：按 `widget-templates.md` 奖励道具框模板撰写，须含 `【需要通用组件图示：通用道具说明tips】` 占位
