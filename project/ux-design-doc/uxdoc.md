## Figma 工作流（严格执行）

> ⚠️ **禁止使用 MCP `download_figma_images`**，其保存路径不符合项目规范。截图一律通过 `figma-capture.py` 导出。

### 脚本用法（在 case 目录下运行，必须加 `-X utf8`）

```bash
# 第一步：列出 Frame 清单（无参数）
python -X utf8 ../scripts/figma-capture.py "<figma_url>"

# 第二步（由用户确认后执行）：下载截图
python -X utf8 ../scripts/figma-capture.py "<figma_url>" 1 3 5

# 读取 Frame 文字结构（不下载图片）
python -X utf8 ../scripts/figma-capture.py "<figma_url>" --text 1 3 5
```

### 标准流程（每个 case 必须按此顺序执行）

1. **用户给出 Figma 链接** → AI 运行列清单命令，输出编号列表
2. **用户确认清单**，明确以下两类：
   - **需要下载图 + 读结构**：将写入交互文档的界面
   - **仅作流程状态说明**：不需要单独下载，文档中只在流程中提及
3. **AI 执行下载**，仅下载用户确认的界面
4. **AI 逐张阅读截图**（`read_file` 查看图片）：
   - ⚠️ 用户明确的所有界面必须全部读完，不可跳过
   - 同一功能的多状态界面必须**对比阅读**，差异点即为交互说明的核心
5. **AI 读取文字结构辅助理解**（运行 `--text`）：
   - 补充截图中不易读取的文字层信息
   - 如截图已能完整理解界面，可跳过，但需主动判断

### 截图保存规范

- 所有截图存入 **`case目录/design/`** 子目录
- 文件名由脚本自动生成（`序号-Frame名.png`），不要手动重命名
- `structure.json` 自动维护，重导时复用文件名避免 HTML 中断图

## 策划文档读取规范

> ⚠️ **禁止为「读文件」这类单步操作派遣 subagent**，延迟高，作为下策。

| 文件格式 | 读取方式 |
|---------|---------|
| `.docx` | `python -X utf8 -c "import docx; doc=docx.Document(r'路径'); [print(p.text) for p in doc.paragraphs if p.text.strip()]"` |
| `.txt` / `.md` / `.json` | 直接用 `read_file` 工具 |

## 参考源优先级规范

### 策划文档（PRD）重点关注
- ✅ 功能逻辑（触发条件、状态判断、边界情况）
- ✅ 交互规则（各种情况下的系统行为）

### 交互稿（Figma）重点关注
- ✅ 界面版式与框架结构
- ✅ 信息层级与排布
- ✅ 控件类型与命名

### 两者不一致时
**以交互稿为准**，PRD 可能更新不及时。发现明确逻辑矛盾时用 `**@策划**` 标注：
```
**@策划** 交互稿中 xxx，PRD 描述为 yyy，请确认以哪个为准。
```

## 写作流程（严格执行）

### 第零步：确定界面排列顺序（不可跳过）

1. **语义强连接的界面必须相邻排列**：同一主界面的多个状态必须紧挨在一起
2. **推荐排列策略（优先级依次降低）**：
   - 主流程顺序：按用户操作路径从头到尾排列
   - 状态聚合：同一界面的多状态变体紧挨排在第一次出现之后
   - 弹窗/侧窗跟随其触发界面
3. **判断标准**：两个界面若互相引用，通常应相邻

### 第一步：建立控件清单（不可跳过）

读完截图、`--text` 结构、PRD 后，**不直接开写**，先在对话中列出每个界面的控件及类型：
```
[2] 活动主界面
- 返回按钮 → 类型：按钮（图标按钮）
- 阶段标签 → 类型：横向 Tab 导航
- 任务面板 → 类型：列表控件
```

### 第一步 B：识别子控件（不可跳过）

对清单中每个「复合控件 / 卡片 / 列表」，**从截图直接识别**内部子控件类型：
```
- 任务面板 → 类型：列表控件
    子控件：按钮（领取）/ 图标按钮（跳转）
```
> ⚠️ 识别依据是截图，不是推断。
> ⚠️ **完成后必须在对话中等待用户确认清单**，确认后再进入第二步。

### 第二步：按模板逐控件写作

参照 `specs/widget-templates.md` 对应条目，以模板为骨架填写，不凭记忆自由发挥。

> ⚠️ 子内容中的控件同样适用模板，不可一笔带过
> ⚠️ 第一步B确认的子控件列表 → 对应模块必须写 `**子控件：** 类型A / 类型B` 字段

### 第三步：validate 验证（不可跳过）

```bash
node ../scripts/validate.mjs interaction-spec.md
```

有 warning 必须对照截图判断是否真实问题，修复后再生成 HTML。

## 规范查阅路径

> ⚠️ **写 case 前必须主动读取以下三个文件，无需用户提醒：**

| 查阅时机 | 文件 | 内容 |
|---------|------|------|
| 撰写前 / 有结构疑问时 | `specs/writing-guide.md` | 扁平结构原则、字段顺序、状态写法、术语统一、P0/P1/P2 分级 |
| 遇到具体控件时 | `specs/widget-templates.md` | 每类控件必须写什么 / 模板 / 检查清单 |
| 字段格式有疑问时 | `specs/field-reference.md` | 字段格式、HTML 样式、常见错误对照表 |

## 交互说明格式速查

| 层级 | 格式 | 说明 |
|------|------|------|
| 文档头 | `icon:` / `version:` / `title:` / `planner:` / `uxd:` | 文件顶部5行，顺序固定 |
| 需求范围 | `## A. 需求范围` | 列出所有界面名称 |
| 方案说明 | `## B. 方案说明` | 按界面分块 |
| 界面块 | `### [N] 界面名称` + `design: 路径` | 方括号整数，截图紧跟其后 |
| 模块 | `#### N. 名称` | 界面内从 1 起编，不跨界面连续 |
| 子模块 | `##### N 名称` | 纯数字无点，内嵌父卡片 |
| 控件类型 | `**类型：** xxx` | 每个模块必填，bold + 全角冒号 + 同行 |
| 控件状态 | `**控件状态：** A / B / C` | `/` 分隔 |
| @职能标注 | `**@策划** xxx` / `**@GUI** xxx` / `**@程序** xxx` | bold，无冒号 |

**@职能颜色**：策划 = 蓝 ｜ GUI / VX = 绿 ｜ 程序 = 红

## 解析器支持的 MD 格式模式

> ⚠️ **只有下列格式能被 `md2html.mjs` 正确识别，其他写法会被静默丢弃。**

| 用途 | 识别的 MD 格式 |
|------|-------------|
| 文档头 | `key: value`（顶部 5 行，无缩进） |
| 界面块 | `### [N] 名称` |
| 界面截图 | `design: 相对路径` |
| 控件类型 | `**类型：** xxx` |
| 显示字段 | `1. 显示：`（单独成行，末尾冒号） |
| 交互字段 | `2. 交互说明：`（单独成行，末尾冒号） |
| 控件状态 | `**控件状态：** A / B / C` |
| @职能 | `**@策划** 文字` / `**@GUI** 文字` / `**@程序** 文字` |
| 子模块 | `##### N 名称` |
| 图示占位（显示） | `【需要图示】` |
| 图示占位（交互） | `【需要交互图示，热区 ≥ 80px】` |
| 图示占位（控件状态） | `【需要控件状态图示】` |
| 图示占位（通用组件） | `【需要通用组件图示：xxx名称】` |

### 常见「静默丢弃」写法（禁止）

| 错误写法 | 正确写法 |
|---------|---------|
| `显示：` 或 `**显示：**` | `1. 显示：` |
| `*类型：* xxx` | `**类型：** xxx` |
| `@策划：xxx` | `**@策划** xxx` |
| **控件状态** 用 `- ` 列表 | `**控件状态：** A / B` 同行 |
| `**触发：** xxx` | `2. 交互说明：` + `触发条件：xxx` 子项 |
